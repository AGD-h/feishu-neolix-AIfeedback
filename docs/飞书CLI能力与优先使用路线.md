# 飞书 CLI 能力与优先使用路线

本文档用于团队理解飞书 CLI 能做什么，以及我们这个「新石器 × 飞书 AI 反馈闭环」项目应该先用哪些能力。

## 一、飞书 CLI 是什么

飞书 CLI（命令名通常是 `lark-cli`）是飞书开放平台提供的命令行工具。它可以让开发者或 AI 助手在终端里操作飞书能力，例如创建云文档、读取文档、更新文档、管理消息、访问多维表格、调用飞书开放平台接口等。

对我们团队来说，它的价值不是替代飞书界面，而是把一些重复动作自动化：

- 把 AI 生成的方案、周报、调研结果直接写入飞书云文档。
- 从飞书多维表格读取工单数据，生成周报。
- 把关键提醒推送到飞书群。
- 让 Cursor / Trae 这类 AI 助手通过 CLI 帮我们操作飞书，而不是只停留在本地代码。

## 二、飞书 CLI 的主要能力

### 1. 云文档能力

可以创建、读取、搜索、更新飞书云文档。

适合我们做：

- 创建开题报告草稿。
- 创建技术实现方案。
- 创建访谈记录模板。
- 创建自动生成的日报、周报。
- 把本地 Markdown 内容导入飞书文档。

典型命令方向：

```bash
lark-cli docs +create
lark-cli docs +fetch
lark-cli docs +update
lark-cli docs +search
```

### 2. 多维表格能力

飞书 CLI 可以配合飞书开放平台访问多维表格数据。多维表格是我们项目的核心，因为它会承载「无人配送反馈工单池」。

适合我们做：

- 读取工单池里的反馈数据。
- 把脚本清洗后的反馈写入工单池。
- 给周报模块提供近 7 天工单数据。
- 批量导入仿真数据或公开舆情数据。

注意：多维表格权限通常需要在飞书开放平台自建应用里开通，并配置 App ID、App Secret、表格 token、table_id。

### 3. 消息通知能力

可以通过飞书消息能力向个人或群发送通知。对于我们项目，这部分主要用于演示「反馈闭环」。

适合我们做：

- P0 安全反馈触发后，自动通知区域负责人。
- 工单超时未处理时，自动提醒群里负责人。
- 每周自动把「反馈聚类周报」推送到团队群。

MVP 阶段可以先用飞书多维表格自动化完成提醒；增强阶段再用 CLI 或机器人 webhook 自动发送。

### 4. 授权与应用配置能力

飞书 CLI 可以帮助配置应用凭证、登录用户授权、检查授权状态。

常见命令方向：

```bash
lark-cli config init --new
lark-cli auth login --recommend
lark-cli auth status
lark-cli doctor
```

我们需要理解两种身份：

- 用户授权：适合创建和编辑你自己的云文档。
- 自建应用凭证：适合脚本读写多维表格、群消息推送等系统能力。

### 5. AI Agent 协作能力

安装飞书 CLI 的 skills 后，AI 助手可以更自然地帮我们操作飞书。例如「把这段内容写成飞书文档」「读取某个文档并总结」「把周报追加到指定文档」。

对我们这种初次开发团队很有用，因为我们可以把繁琐的 API 调用交给 AI 助手处理。

## 三、我们项目应该先用哪些能力

结合我们当前阶段和团队能力，建议按下面顺序来。

### 第一优先级：云文档创建与更新

这是最容易上手、立刻有价值的能力。

先做这些：

- 用 CLI 创建项目说明文档。
- 用 CLI 创建开题报告草稿。
- 用 CLI 创建访谈提纲。
- 用 CLI 创建技术实现方案。

为什么先做它：

- 不需要复杂表格权限。
- 成功反馈最直观，跑完就能得到文档链接。
- 能让我们快速建立「AI 生成内容 → 飞书沉淀」的工作流。

### 第二优先级：多维表格读取

等 2号把「无人配送反馈工单池」建好后，再用 CLI 或飞书开放平台读取表格数据。

先做这些：

- 让 2号提供多维表格 URL、app_token、table_id。
- 在飞书开放平台给自建应用开通多维表格读取权限。
- 让 3号的 `weekly_report.py` 读取近 7 天工单数据。

为什么第二步做它：

- 它直接服务我们的核心功能「聚类周报」。
- 但它比云文档复杂，需要表格权限和应用配置，所以不适合作为第一步。

### 第三优先级：群消息推送

等周报能本地生成后，再把周报推送到飞书群。

先做这些：

- 建一个团队飞书群。
- 配置群机器人 webhook，或用飞书 CLI 消息能力。
- 让脚本把周报摘要发送到群里。

为什么第三步做它：

- 它能增强演示效果。
- 但不是系统核心，先保证文档和表格跑通。

### 第四优先级：自动写入多维表格

等我们有了舆情采集或一车一码页面，再把外部数据自动写入工单池。

先做这些：

- 1号把公开评论清洗成标准 Schema。
- 脚本调用飞书 API 写入多维表格。
- 验证新增记录能触发 2号配置的 AI 分类和自动化流程。

为什么最后做它：

- 自动写入涉及权限、字段映射、API 调用，容易踩坑。
- MVP 可以先用 CSV 导入和飞书表单，不影响演示。

## 四、我们现在的推荐行动路线

### 今天到明天

1. 安装并登录飞书 CLI。
2. 用 CLI 创建第一篇云文档。
3. 确认我们可以从本地 Markdown 生成飞书文档链接。

### 2号建表后

1. 让 2号把多维表格 URL 发给团队。
2. 确认 app_token、table_id 的位置。
3. 准备自建应用权限。

### 周报模块开始时

1. 3号读取多维表格数据。
2. 调 DeepSeek 生成聚类周报。
3. 用飞书 CLI 创建或更新周报云文档。

### 路演前

1. 用 CLI 自动生成最终展示用的日报/周报。
2. 把演示链接统一放进飞书知识库或云文档目录。
3. 形成「反馈进入工单池 → AI 处理 → 飞书沉淀周报」的完整证据链。

## 五、对我们团队的具体建议

### 对 1号 数据线

先不用研究复杂的自动写表 API。你的优先级是：

1. 生成仿真数据 CSV。
2. 整理真实访谈和公开舆情样本。
3. 后续再把清洗后的数据自动写入多维表格。

你先用飞书 CLI 的文档能力沉淀「访谈记录」和「数据分布说明」即可。

### 对 2号 系统线

你主要还是在飞书界面里工作：

1. 建多维表格工单池。
2. 配 AI 字段。
3. 配自动化和仪表盘。

飞书 CLI 对你不是第一优先级。你只需要学会提供表格 URL、app_token、table_id 给 3号。

### 对 3号 输出线

你最应该先学飞书 CLI。

原因：

- 你负责开题报告、周报、日报、路演材料。
- 飞书 CLI 可以让你把 AI 生成内容快速变成云文档。
- 后续 `weekly_report.py` 也需要把周报写入飞书文档。

你的第一目标是：跑通「本地 Markdown → 飞书云文档」。

## 六、我们不要一开始就做的事

暂时不要做这些：

- 不要一上来研究所有飞书 CLI 命令。
- 不要先做复杂服务器部署。
- 不要先做完整自动写入多维表格。
- 不要让三个队员分别配置不同的飞书应用。
- 不要把 App Secret、DeepSeek Key 写进文档或提交 GitHub。

我们先用最小闭环证明能力：

> 本地内容生成 → 飞书云文档沉淀 → 多维表格工单池 → 周报自动生成

## 七、第一条可执行命令

如果飞书 CLI 已经安装并授权，可以先在项目根目录运行：

```powershell
lark-cli docs +create --doc-format markdown --title \"飞书 CLI 能力与优先使用路线\" --content \"@docs/飞书CLI能力与优先使用路线.md\"
```

成功后终端会返回飞书云文档链接，把链接发到团队群即可。
